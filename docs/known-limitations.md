# Known Limitations & Honesty Sheet

> **AeroSentinal — Phase 1 Prototype**
> This document is designed to be shown to mentors proactively. Teams that name their own gaps first earn more trust than teams whose gaps are discovered during questioning.

---

## 1. Data Provenance

| Subsystem | Dataset | Real vs Synthetic | Notes |
|---|---|---|---|
| Engine (RUL) | NASA C-MAPSS FD001/FD004 | **Synthetic** (NASA-generated) | Industry-standard benchmark. C-MAPSS is itself a simulated dataset (Commercial Modular Aero-Propulsion System Simulation). No team in PHM uses real airline degradation data publicly. |
| Hydraulics | UCI Hydraulic Condition Monitoring | **Real sensor data** | 2205 measurement cycles from a real hydraulic test rig. Four fault types with known severity levels. |
| Landing Gear | Physics-informed generator | **Synthetic** | No public dataset for brake wear telemetry. Generator uses kinetic energy model with mass, speed, deceleration. |
| APU | Physics-informed generator | **Synthetic** | No public degraded-APU EGT dataset. Generator models startup curve degradation with controllable fouling. |
| ECS | Reverse-Brayton-cycle simulator | **Synthetic** | By design — the ECS module is a physics simulator, not an ML model. Parameterized per thermodynamic first principles. |

**Key disclosure:** The C-MAPSS dataset, while industry-standard, is itself synthetic (generated by NASA's simulation). We use it because it's the most rigorous publicly available benchmark for turbofan RUL prediction, and our results are directly comparable to published literature.

## 2. Model Status

| Subsystem | Target Model | Current Status | What's Real | What's Simulated |
|---|---|---|---|---|
| Engine | BiLSTM + Attention → ONNX | **API live, simulation mode** | Service architecture, API contract, response schema | Predictions are from a degradation curve, not a trained model |
| Hydraulics | 1D Conv Autoencoder → ONNX | **API live, simulation mode** | Service architecture, reconstruction error concept | Error values are randomized, not from trained autoencoder |
| Landing Gear | XGBoost Classifier → ONNX | **API live, simulation mode** | Severity classification logic | Classification is rule-based, not ML |
| APU | Random Forest → ONNX | **API live, simulation mode** | EGT margin calculation | Deviation scoring is parametric, not RF |
| ECS | Physics model (no ML) | **API live, production** | Thermodynamic simulation is the actual model | N/A — this is intentionally physics-based |

**Every API response includes `is_synthetic_data: true/false`** so the UI never misrepresents simulated outputs as trained model predictions.

## 3. What Works End-to-End (Defensible in Demo)

- **Cross-domain coupling attribution**: Inject ECS fouling → observe false engine health depression → fusion layer correctly attributes root cause. This is live, testable, and demonstrates the architecture's core differentiator.
- **ACARS 220-character compiler**: Real compression from full diagnostic JSON to ACARS-format message. ~12x compression ratio measured.
- **AOG risk scorer**: Wired to live model outputs. `P(failure) × $150K/hr × grounding_hours` — not hardcoded.
- **What-if simulator**: Re-runs models live with injected faults. Not replaying a canned animation.
- **NLP retrieval**: TF-IDF + cosine similarity over 18 curated maintenance pairs. Works, but is explicitly a placeholder.

## 5. Known Technical Debt

1. **No WebSocket streaming**: The frontend polls every 10s instead of receiving push updates. Adequate for prototype, not for production.
2. **No authentication/authorization**: The API is open. Production deployment requires JWT + role-based access.
3. **No database**: All state is in-memory. Production needs PostgreSQL + time-series DB (InfluxDB/TimescaleDB).
4. **No CI/CD pipeline**: Manual deployment. GitHub Actions should be configured for automated testing.
5. **No model versioning**: When trained models are deployed, MLflow model registry should track versions.
6. **CPU-only inference**: No GPU acceleration configured. PyTorch uses CPU. Acceptable for prototype.

## 7. Honest Framing for Mentor Presentation

**Lead with strengths:**
- "We built the full architecture with 5 subsystem APIs, a fusion layer, and a cross-domain coupling demonstrator"
- "Every response tells you whether the data is real or simulated"
- "The ECS→Engine coupling attribution works live — try the simulator"

**Be upfront about gaps:**
- "ML model training is in progress by our team member"
- "The NLP assistant uses TF-IDF retrieval, not a fine-tuned LLM — that's Phase 2"
- "We haven't measured real inference latency yet because we need trained ONNX models first"

**Never say:**
- "Our model predicts with X% accuracy" (unless you have the trained model's actual test metrics)
- "This runs on edge hardware" (unless you've actually deployed to a Jetson/similar)
- "This is production-ready" (it's a prototype, say so)
